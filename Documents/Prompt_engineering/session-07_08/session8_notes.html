Reality Check

The bot doesn't know loan BFL2024001. It has never seen it. Your database is not connected. The LLM is just predicting plausible-sounding numbers based on patterns in its training data. ₹8,450? Made up. 22 EMIs remaining? Made up. This is called hallucination.

Why This Happens — Three Root Causes

📅
Cause 1

Knowledge Cutoff

gpt-4o-mini was trained on data up to a certain date. It has never seen your Bajaj Finance database. Can't access customer data it was never trained on.

🧮
Cause 2

Can't Calculate

LLMs don't do math — they predict the next likely word. Ask it 2+2=? It predicts "4" because that's what it saw during training. Not calculated — predicted.

🚫
Cause 3

Can't Take Action

LLMs only generate text. They can't call your refund API, update your database, or send an SMS. They're a brain without hands.

The Metaphor That Fixes Everything

An LLM is like a brilliant student locked in a library with books from 2024.

  • Ask about history → confident, correct answers (it's in the books)
  • Ask about today's stock price → confident, made-up answers (not in the books)
  • Ask it to buy something → can't (no phone, no internet, no hands)

What does this student need? Tools. A phone. Internet access. A calculator. A way to fetch live data. That's what we're building today.

Session 08 — Chapter 4 of 10

The Tool Card.

Before writing any code, understand what a "Tool" actually looks like to the LLM. It's not magic — it's a structured description with 4 specific parts. Get these right and everything works.

When you give a tool to the LLM, what you're actually giving it is a Tool Card — a structured description. The LLM reads this card to decide whether to use the tool. Four fields. That's it.

📋 Tool Card Anatomy
① NAME
Unique identifier. Snake_case. Descriptive. get_loan_status
② DESCRIPTION
The most important field. Tells LLM when to use this tool. Be clear and specific.
③ PARAMETERS
Schema of inputs needed — name, type, required/optional. Defined via Pydantic in LangChain.
④ OUTPUT (implicit)
What the tool returns. LangChain infers this from your function's return value.

Building a Tool Card — Bajaj Finance Example

Let's design the get_loan_status tool card. This is the mental model — in the next chapter we'll write actual code.

tool_card_conceptual.pyCONCEPT
# ① NAME — what the LLM will call
name = "get_loan_status"

# ② DESCRIPTION — when and why to use it (the LLM's decision-maker)
description = """Fetches the current status of a Bajaj Finance loan including
EMI amount, remaining tenure, total outstanding, and next due date.
Use this when the customer asks about their loan details, EMI,
balance, or tenure."""

# ③ PARAMETERS — what inputs it needs
parameters = {
    "loan_id": {
        "type": "string",
        "description": "The loan account number, format BFL followed by digits (e.g. BFL2024001)",
        "required": True
    }
}

# ④ OUTPUT — what it returns (structured data)
output_example = {
    "customer_name": "Rahul Sharma",
    "loan_type": "Personal Loan",
    "emi": 8450,
    "remaining_months": 22,

The Critical Insight — Tool Card ≠ Tool

The Menu vs The Kitchen

When you walk into a restaurant, you don't see the kitchen. You see the menu. The menu tells you:

  • What dishes are available (name)
  • What's in each dish (description)
  • What you need to specify (parameters — "how spicy?", "extra cheese?")

The actual cooking happens in the kitchen — which you never see. The LLM is the customer. The Tool Card is the menu. Your Python function is the kitchen. The LLM reads the menu to order — your code runs the kitchen to cook.

Why Description Matters Most

❌ Bad Description

"A tool that gets loan data."

LLM doesn't know when to use it. Misses obvious cases like "What's my EMI?"

✅ Good Description

"Fetches EMI amount, remaining tenure, outstanding balance for a Bajaj loan. Use when customer asks about loan details, EMI, balance, or tenure."

LLM knows exactly when this matches customer intent.

Golden Rule

The LLM decides which tool to use by reading descriptions. If your description is weak, the LLM won't pick your tool even when it's the perfect match. Spend more time writing the description than any other part.

Session 08 — Chapter 5 of 10

The 5-step lifecycle.

Here's what actually happens from the moment a customer says "What's my EMI?" to the moment they see the answer. Understanding this flow is the difference between "it works" and "I know why it works".

Follow This Example Throughout

Customer query: "My loan is BFL2024001. What's my current EMI?"

We'll trace this query through all 5 steps. By the end, you'll know exactly what the LLM does, what your code does, and where the responsibility split is.

1

LLM detects intent + finds information gap

The LLM reads the customer's message. Its internal reasoning: "They want EMI info for a specific loan. I don't have that data in my training. But I have a tool — get_loan_status — that does exactly this."

Internal reasoning (invisible to you):
Intent: retrieve loan data
Gap: no database access in training
Matching tool: get_loan_status ✓
2

LLM generates a structured tool call (JSON)

The LLM doesn't execute code. It generates a structured request — a JSON object saying "call this tool with these arguments". That's all it does.

{
  "tool": "get_loan_status",
  "args": { "loan_id": "BFL2024001" }
}
3

Your backend executes the function

Your code reads the tool call, finds the Python function named get_loan_status, extracts the arguments, and calls it manually. The function runs — queries the database, returns real data.

Your code runs:
result = get_loan_status(loan_id="BFL2024001")
# → queries bajaj_db.json
# → returns real loan data
4

Tool result sent back to LLM

LangChain takes the function's return value and sends it back to the LLM as a tool message. The LLM now has real data to work with.

{
  "emi": 8450,
  "remaining_months": 22,
  "outstanding": 185900
}
5

LLM synthesizes a human response

The LLM converts the structured data into natural language for the customer. This is where personality, tone, and formatting come in.

Final reply to customer:
"Your EMI for BFL2024001 is ₹8,450 per month. You have 22 EMIs remaining with an outstanding balance of ₹1,85,900."

The Critical Division of Labor

🧠 LLM DOES
  • Understand intent
  • Decide which tool to use
  • Format the arguments
  • Generate final human response
💻 YOUR CODE DOES
  • Run the actual function
  • Query the database
  • Call external APIs
  • Return structured data to LLM
Why This Split?

Security + Control. If LLMs could execute code directly, it would be terrifying — a hallucinating AI running arbitrary Python on your server? No thanks. The LLM suggests, your code validates and executes. You stay in control of every real-world action.

In This Session, You'll Do Every Step Manually

From Chapter 6 onwards, you'll write code for every one of these 5 steps yourself. LLM generates tool call → you parse it → you execute the function → you send result back → LLM synthesizes. No hidden magic. This way you understand exactly what's happening under the hood.

Step 2 — Write the Tool with @tool

🔧 Create your first LangChain tool
Goal: a function the LLM can call to fetch loan data
bajaj_tools.pyBUILD
from langchain_core.tools import tool
import json

# Load our mock database once at startup
with open("bajaj_db.json") as f:
    db = json.load(f)

# The @tool decorator is LangChain magic
# It reads the function signature + docstring → builds the Tool Card automatically
@tool
def get_loan_status(loan_id: str) -> dict:
    """Fetches current loan status from Bajaj Finance database.
    
    Returns EMI amount, remaining tenure, outstanding balance,
    and next due date for the given loan account.
    
    Use this when the customer asks about their loan details,
    EMI, balance, or tenure.
    
    Args:
        loan_id: The loan account number (e.g., 'BFL2024001')
    """
    if loan_id not in db["loans"]:
        return {"error": f"Loan {loan_id} not found"}
    
    loan = db["loans"][loan_id]
    return {
        "customer_name": loan["customer_name"],
        "loan_type": loan["loan_type"],
        "emi": loan["emi"],
        "remaining_months": loan["remaining_months"],
        "outstanding": loan["outstanding"],
        "next_due_date": loan["next_due_date"]
    }
What @tool Does Automatically

LangChain's @tool decorator reads your function and automatically builds the Tool Card for the LLM:

  • name = function name (get_loan_status)
  • description = docstring (the """...""" text)
  • parameters = function arguments + type hints (loan_id: str)

No separate JSON schema. No manual parsing. Your Python function IS the tool definition. This is why LangChain is so much cleaner than plain Gemini/OpenAI SDKs for learners.

Why This is "Sequential"

The Dependency Chain

Tool 2 needs the output of Tool 1 — specifically outstanding, interest_rate, and prepayment_charge_pct. You can't call them at the same time. The order matters:

  • Call get_loan_status first → wait for result
  • Pass result's data to calculate_prepayment_impact

LangChain's agent handles this automatically — it sees that Tool 2 needs data from Tool 1 and calls them in order.

Build the Calculation Tool

🧮 Add calculate_prepayment_impact to bajaj_tools.py
Goal: a pure computation tool (no database access — just math)
bajaj_tools.py (add this)BUILD
@tool
def calculate_prepayment_impact(
    outstanding: float,
    prepay_amount: float,
    interest_rate: float,
    charge_pct: float
) -> dict:
    """Calculates prepayment penalty and interest savings for a loan.
    
    Use when customer wants to know the financial impact of making
    a partial prepayment on their loan.
    
    Args:
        outstanding: Current outstanding principal (₹)
        prepay_amount: Amount customer wants to prepay (₹)
        interest_rate: Annual interest rate (%)
        charge_pct: Prepayment charge percentage (%)
    """
    # Penalty = percentage of prepay amount
    penalty = (prepay_amount * charge_pct) / 100
    
    # Simplified interest saving = prepay × rate × avg remaining tenure (2 yrs)
    interest_saved = (prepay_amount * interest_rate * 2) / 100
    
    net_benefit = interest_saved - penalty
    
    return {
        "penalty": round(penalty, 2),
        "interest_saved": round(interest_saved, 2),
        "net_benefit": round(net_benefit, 2),
        "recommendation": "Prepayment beneficial" if net_benefit > 0 else "Not worth it"
    }

🔗 Tool 1 runs → take its output → feed to Tool 2

sequential_manual.py
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, ToolMessage
from bajaj_tools import get_loan_status, calculate_prepayment_impact

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# Give LLM BOTH tools — it will decide which to call first
tools = [get_loan_status, calculate_prepayment_impact]
llm_with_tools = llm.bind_tools(tools)

# Helper: map tool name → tool function (for easy lookup)
tool_map = {"get_loan_status": get_loan_status, 
            "calculate_prepayment_impact": calculate_prepayment_impact}

user_question = "For loan BFL2024001, if I prepay ₹50,000, what's the penalty and savings?"
messages = [HumanMessage(content=user_question)]

# ─── TURN 1: LLM decides to call get_loan_status first ───
response = llm_with_tools.invoke(messages)
print("Turn 1 — LLM wants to call:", [tc["name"] for tc in response.tool_calls])
messages.append(response)

# Execute whatever LLM asked for
for tc in response.tool_calls:
    result = tool_map[tc["name"]].invoke(tc["args"])
    messages.append(ToolMessage(content=str(result), tool_call_id=tc["id"]))

# ─── TURN 2: LLM sees loan data, now calls calculate_prepayment_impact ───
response = llm_with_tools.invoke(messages)
print("Turn 2 — LLM wants to call:", [tc["name"] for tc in response.tool_calls])
messages.append(response)

for tc in response.tool_calls:
    result = tool_map[tc["name"]].invoke(tc["args"])
    messages.append(ToolMessage(content=str(result), tool_call_id=tc["id"]))

# ─── TURN 3: LLM has all data, writes final reply ───
final = llm_with_tools.invoke(messages)
print("\nFinal answer:")
print(final.content)

The Loop Pattern You Just Wrote

Notice the shape — call LLM, execute tools, append results, call LLM again. Repeat until LLM stops asking for tools. This loop is the heart of function calling:

# The pattern in pseudocode:
while LLM has tool_calls:
    execute each tool_call
    append ToolMessage results to messages
    call LLM again # ← LLM sees results, decides next step

# When LLM stops asking for tools → final text answer
Why This is "Sequential Reasoning"

The LLM figured out the order by itself — it knew calculate_prepayment_impact needed outstanding and interest_rate, which come from get_loan_status. So it called them in the right order. You just looped and executed.

Session 08 — Chapter 8 of 10

Three tools,
at the same time.

When tools are independent — none needs another's output — don't waste time calling them sequentially. Run them in parallel. Customer sees the result 3x faster.

Sequential vs Parallel — The Difference

❌ Sequential (slow)

Tool 1 (2s) → Tool 2 (2s) → Tool 3 (2s)

Total: 6 seconds

✅ Parallel (fast)

Tool 1 (2s) ║ Tool 2 (2s) ║ Tool 3 (2s)

Total: ~2 seconds

When to Use Parallel — The Rule

The Independence Test

Before running tools in parallel, ask: does any tool need output from another tool?

  • Yes → sequential (prepayment needs loan data first)
  • No → parallel (weather, hotels, flights for a trip are independent)

Today's example: a customer asks for loan status + next EMI due date + recent refund requests. Three independent queries — perfect for parallel execution.

@tool
def get_emi_schedule(loan_id: str, months_ahead: int = 3) -> dict:
    """Returns upcoming EMI schedule for a loan.
    
    Use when customer asks about future EMI dates or upcoming payments.
    """
    if loan_id not in db["loans"]:
        return {"error": "Loan not found"}
    loan = db["loans"][loan_id]
    return {
        "loan_id": loan_id,
        "emi_amount": loan["emi"],
        "upcoming_dates": ["2026-05-05", "2026-06-05", "2026-07-05"][:months_ahead]
    }

Goal: LLM returns 3 tool calls in one response, you execute all of them
parallel_manual.py
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, ToolMessage
from bajaj_tools import get_loan_status, get_emi_schedule, check_interest_rate

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

tools = [get_loan_status, get_emi_schedule, check_interest_rate]
llm_with_tools = llm.bind_tools(tools)

tool_map = {
    "get_loan_status": get_loan_status,
    "get_emi_schedule": get_emi_schedule,
    "check_interest_rate": check_interest_rate,
}

# Query that needs THREE INDEPENDENT pieces of info
user_question = "For loan BFL2024001, show me: current status, next 3 EMI dates, " \
                "and current personal loan rate at Bajaj."
messages = [HumanMessage(content=user_question)]

# ─── LLM returns MULTIPLE tool_calls in ONE response ───
response = llm_with_tools.invoke(messages)
print(f"LLM asked for {len(response.tool_calls)} tools:")
for tc in response.tool_calls:
    print(f"  → {tc['name']}({tc['args']})")

messages.append(response)

# ─── Execute ALL of them (loop — fast since all are independent) ───
for tc in response.tool_calls:
    result = tool_map[tc["name"]].invoke(tc["args"])
    messages.append(ToolMessage(content=str(result), tool_call_id=tc["id"]))

# ─── LLM synthesizes final answer from all 3 results ───
final = llm_with_tools.invoke(messages)
print("\nFinal answer:")
print(final.content)
LLM asked for 3 tools: → get_loan_status({'loan_id': 'BFL2024001'}) → get_emi_schedule({'loan_id': 'BFL2024001', 'months_ahead': 3}) → check_interest_rate({'loan_type': 'personal loan'}) Final answer: Here's the info for loan BFL2024001: • Current EMI: ₹8,450/month, 22 EMIs remaining • Next 3 EMI dates: 5 May, 5 June, 5 July 2026 • Current personal loan rate: 11.5% p.a.
Sequential vs Parallel — Same Loop, Different Behavior

Look closely — the code in Ch.7 (Sequential) and Ch.8 (Parallel) is the same loop pattern. What changes?

  • Sequential (Ch.7): LLM returns 1 tool_call per turn → you loop multiple turns
  • Parallel (Ch.8): LLM returns 3 tool_calls in 1 turn → you execute all in one loop

The LLM decides the shape based on whether tools depend on each other. You just execute what it asks.

Speed Improvement

Sequential would need 3 separate LLM round-trips: ~3 seconds. Parallel = 1 LLM round-trip + 3 fast function calls: ~1 second. For a customer-facing chatbot, this is the difference between "snappy" and "slow".

Why gpt-4o-mini is Good at This

Modern tool-calling models like gpt-4o-mini are trained to detect when multiple tools are independent and return them together in one response. Older models returned one tool call at a time, forcing sequential execution.

When tools fail.

Tools fail in production — APIs timeout, databases go down, inputs are malformed. A production bot handles failures gracefully. Let's see the 5 strategies.

What Could Go Wrong?

❓

Invalid Input

The 5 Strategies

Strategy 1 — Isolate Failures

If one tool fails in a multi-tool query, don't crash everything. Return what worked.

@tool
def get_loan_status(loan_id: str) -> dict:
    """Fetches loan status. Returns structured error if loan not found."""
    try:
        if loan_id not in db["loans"]:
            return {"error": f"Loan {loan_id} not found in our system"}
        return db["loans"][loan_id]
    except Exception as e:
        # Don't raise — return structured error so LLM can explain to user
        return {"error": f"Database error: {str(e)}"}

Strategy 2 — Use Timeouts

from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage, HumanMessage, ToolMessage
from bajaj_tools import (
    get_loan_status,
    calculate_prepayment_impact,
    get_emi_schedule,
    check_interest_rate,
)

# ─── 1. Setup: LLM + Tools ───
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

tools = [get_loan_status, calculate_prepayment_impact,
         get_emi_schedule, check_interest_rate]
llm_with_tools = llm.bind_tools(tools)

tool_map = {t.name: t for t in tools}

# ─── 2. System prompt (personality from S7) ───
system_prompt = """You are a professional Bajaj Finance customer support agent.
- ALWAYS use tools to fetch real data — NEVER make up numbers
- If a tool returns an error, explain it helpfully
- Never mention competitor banks
- Keep responses under 4 sentences"""

# ─── 3. Memory: a simple list of messages (same idea as S7) ───
conversation = [SystemMessage(content=system_prompt)]

# ─── 4. The chat function — the reusable loop ───
def chat(user_message: str) -> str:
    """Send user message, loop until LLM gives final text answer."""
    # Add user message to memory
    conversation.append(HumanMessage(content=user_message))
    
    # Loop: keep calling LLM until it stops asking for tools
    while True:
        response = llm_with_tools.invoke(conversation)
        conversation.append(response)
        
        # If no tool calls, LLM is done — return its text answer
        if not response.tool_calls:
            return response.content
        
        # Otherwise, execute each requested tool call
        for tc in response.tool_calls:
            result = tool_map[tc["name"]].invoke(tc["args"])
            conversation.append(ToolMessage(
                content=str(result),
                tool_call_id=tc["id"]
            ))

# ─── 5. USE IT — multi-turn conversation ───
turns = [
    "Hi, my loan is BFL2024001. What's my current EMI?",
    "How many are left?",                                       # ← memory
    "If I prepay ₹50,000 next month, what's the net benefit?",  # ← 2 tools
    "What's the current personal loan rate at Bajaj?",         # ← another tool
]

for i, turn in enumerate(turns, 1):
    print(f"\n[Turn {i}]")
    print(f"Customer: {turn}")
    print(f"Bot: {chat(turn)}")
Read the chat() Function Carefully

This 15-line function is the core of every tool-using chatbot. It's literally:

  1. Add user message to conversation
  2. Ask LLM
  3. If LLM wants tools → execute them, add results, loop back to step 2
  4. If LLM gives text → return the answer

Memory works because we append everything to conversation — same list passed in every invoke(). Same idea as S7, just with tools added.

Q1. Your S7 bot confidently tells a customer their EMI is ₹8,450 — but there's no database connection. What is this called?
When LLMs don't have real data, they predict what seems likely based on patterns in training. The output is confident but factually wrong — the core problem that tools solve.
Q2. Which tool type would calculate_prepayment_impact fall under?
Computation tools take inputs, run reliable math (not LLM prediction), and return results. They solve the "LLMs can't do math reliably" problem.
Q3. In the Tool Card, which field does the LLM use MOST to decide whether to call a specific tool?
The description is the primary signal for tool selection. A bad description = LLM won't pick the tool even if it's perfect for the query. The name and parameters are secondary.
Q4. In LangChain, how do you turn a Python function into a tool?
@tool is LangChain's magic — it reads your function name (→ tool name), docstring (→ description), and type-hinted parameters (→ schema). Zero boilerplate.
Q5. What does the LLM actually do when it "calls a function"?
LLMs can't execute code. They generate a structured request (the tool_calls you saw in Ch.6). Your backend code parses that request and runs the actual function. This security separation is why function calling is safe.
Q6. Customer asks: "For loan BFL2024001, if I prepay ₹50,000, what's the benefit?" Your bot needs get_loan_status AND calculate_prepayment_impact. How should they execute?
calculate_prepayment_impact needs outstanding, interest_rate, charge_pct — which come from get_loan_status. Sequential is required when there's a dependency. The LLM detects this and calls them one after another.
Q7. When should you use parallel tool calling?
The test is independence. Weather + hotels + flights for a trip? Independent → parallel. Loan status + prepayment calc? Dependent → sequential.
Q8. A customer asks about loan BFL9999 but it doesn't exist in your database. What's the best way to handle this?
Structured errors let the LLM craft helpful user-facing messages like "I couldn't find that loan — can you double-check the number?". Hiding errors misleads users. Crashing is worse.
Q9. In your manual chat() function (Ch.10), why do you use a while loop instead of calling the LLM just once?
For a single-tool query, 1 iteration is enough. But for sequential chaining (prepayment needs loan data first), the LLM returns tool calls, you execute them, then call LLM again → it sees new data → requests the next tool. The loop ends only when the LLM gives a final text answer (no more tool_calls).
Q10. You set temperature=0 on your Bajaj Finance bot. Why?
For finance/health/legal bots, you want predictable, consistent tool usage — never creative surprises. temperature=0 means the LLM picks the most likely tool every time. For creative writing bots, higher temperature is fine.
