Session 07 — Chapter 1 of 9

One prompt.
One reply.
Now what?

Your Session 6 LangChain chain was perfect for single-turn tasks. But the moment a customer asks a follow-up question — everything breaks. Let's see exactly why.

In Session 6, you built: chain = prompt | llm | parser. One input → one output. It worked beautifully for the email classifier. But that's a single-turn task — one email, one classification. A real helpdesk chatbot needs multi-turn conversations. That's a completely different problem.

Our Use Case — All Session

🏦 Bajaj Finance Helpdesk Chatbot. 80 million customers. Daily queries about loan balances, EMI schedules, prepayment charges, foreclosure documents. A customer calls in:

  • Turn 1: "My loan is BFL2024001. My EMI of ₹8,450 was deducted twice."
  • Turn 2: "How many EMIs are left?"
  • Turn 3: "Can I prepay ₹50,000 next month?"
  • Turn 4: "What's the penalty for that?"

This is a perfectly natural 4-turn conversation. With Session 6's chain — it breaks after Turn 1. This entire session fixes that.

Problem 1 — The Bot Has Zero Memory

Every time you call chain.invoke() or llm.invoke(), it is a completely fresh, isolated API request. The model has never seen any previous call. It's like calling a new agent every single time.

Bajaj Bot — Plain chain.invoke() (Session 6 approach)
Turn 1 — works fine

AI

Turn 2 — completely separate invoke() call, no context passed

How many EMIs are left?
AI
I don't have access to your loan details. Could you please provide your loan account number so I can assist you?
Root Cause

Turn 1 and Turn 2 are two completely separate HTTP requests to the OpenAI API. The model that handled Turn 1 is gone. A fresh model picks up Turn 2. It has never seen "BFL2024001" or "₹8,450". So "How many EMIs are left?" has no meaning — left for which loan?

This is called being stateless — the model holds zero state between calls.

The Solution

Chat Completion API + LangChain Memory. Chat Completion introduces structured, role-based messages that solve all 3 problems: persistent context, role separation, and behavioral control — all in one clean API.

Session 07 — Chapter 2 of 9

Chat Completion API

Not magic — just structured prompt engineering. Instead of one flat string, you send a labeled list of messages. Each message says who said it and what they said. That's the entire concept.

Chat Completion is an API design pattern that takes a list of role-tagged messages as input and generates the next assistant response. It turns a text-prediction engine into something that behaves like a real conversation.

The Key Mental Model

The LLM hasn't changed. It's still the same gpt-4o-mini predicting the next token. What changed is how you structure the input. Instead of: "Answer this question: ...", you now send a structured list where every message is labeled with its role — who said it — and the model uses that structure to understand the conversation.

Before vs After — What Changes

before_after.py
# ───── SESSION 6 WAY (plain invoke) ─────
result = chain.invoke({"subject": "...", "body": "..."})
# One dict in → one string out → no memory

# ───── SESSION 7 WAY (chat completion) ─────
messages = [
    {"role": "system",    "content": "You are a Bajaj Finance agent..."},
    {"role": "user",      "content": "My loan BFL2024001. EMI deducted twice."},
    {"role": "assistant", "content": "Logged as High Priority billing issue."},
    {"role": "user",      "content": "How many EMIs are left?"},
    # ↑ model sees ALL previous turns — knows exactly what "EMIs" refers to
]

The Structure — Just 2 Fields Per Message

Every single message in a Chat Completion request has exactly two fields. Nothing more:

message_structure.py
{
    "role"    : "system" | "user" | "assistant",  # WHO is speaking
    "content" : "Any text..."                        # WHAT they're saying
}

# That's the entire Chat Completion message format.
# Two fields. That's it. Everything builds on these two fields.
What This Solves
  • Memory: All previous turns are in the list. The model sees full context every call.
  • Role separation: Model clearly knows what's an instruction vs a query vs its own past reply.
  • Behavior control: System message sets rules once — applies to every response automatically.
  • No manual stitching: You append messages to a list. The SDK handles the rest.
Session 07 — Chapter 3 of 9

3 roles.
Everything runs on them.

System. User. Assistant. Three labels. They define who is speaking — and that changes how the model interprets every single word in a message.

Bajaj Finance Call Center Analogy

Before an agent takes their first customer call, the manager briefs them privately: "Be formal. Escalate billing disputes. Never quote competitor rates."

They never repeat this script to customers — but every response they give follows it. That briefing is the system role.

Then a customer calls — that's user. The agent replies — that's assistant. Three roles. Simple.

system

🎬 The Director

Sets the assistant's persona, tone, rules, and domain expertise. Written once at the start by you (the developer). Never shown to the customer. Silently shapes every single response.

"You are a Bajaj Finance support agent. Always be formal. Never mention competitor banks. Escalate billing disputes to Level 2."
user

👤 The Customer

Every message from the customer — questions, requests, information they provide. The assistant reads every user message as something it needs to respond to.

"My loan is BFL2024001. The EMI of ₹8,450 was deducted twice this month."
assistant

🤖 The Agent

The model's generated replies. You don't write these from scratch — they're generated. But you re-send them in every subsequent request as part of the message history. That's what creates memory.

"I've logged loan BFL2024001 as a High Priority EMI dispute. Refund will be processed within 24 hours."

Why Role Separation Matters

Without roles, the model sees one flat blob of text and has to guess what's an instruction, what's a customer query, and what's its own previous response. With roles, everything is unambiguous.

roles_explained.py
# system: behavioral rule — written by YOU, applies to ALL turns
{"role": "system", "content": "You are a Bajaj Finance support agent. Be concise."}

# user: what the customer says
{"role": "user",   "content": "Loan BFL2024001 — EMI ₹8,450 deducted twice."}

# assistant: what the model replied (previous turn — re-sent for memory)
{"role": "assistant", "content": "Logged as High Priority. Refund in 24 hours."}

# user: follow-up — the model knows "EMIs" = BFL2024001 from above
{"role": "user",   "content": "How many EMIs are left?"}
# ↑ No ambiguity. The model sees a structured conversation, not a blob.
Only 3 Roles — Why?

That's genuinely all you need. System = instructions. User = input. Assistant = output. Every possible interaction in a conversation fits into one of these three. Simple enough for the model to reason about clearly. Powerful enough to build production chatbots.

Session 07 — Chapter 4 of 9

The system prompt.
Your secret weapon.

Write it once. Every response for the entire session follows it. The customer never sees it. But it controls everything — tone, format, constraints, domain expertise.

The system prompt is the invisible instruction manual that tells the assistant who it should be and how it should behave — before the first customer message is even read. It's the single most powerful tool in conversational AI.

The 5 Components of a Great System Prompt

1

Identity — who is the assistant?

Give it a specific role. Not just "you are a helpful assistant" — that's too generic. "You are a professional Bajaj Finance customer support agent with 5 years of experience in loan management."

2

Task scope — what can and cannot it do?

What topics should it handle? What should it refuse or escalate? "Handle EMI queries, prepayment, and foreclosure. Escalate legal threats to Level 2."

3

Output format — how should it respond?

Prose? Bullet points? JSON? Short or long? "Keep responses under 3 sentences. Always confirm the loan account number in your reply."

4

Constraints — what rules must it follow?

Brand rules, legal constraints, tone requirements. "Never suggest competitor financial products. Never promise refunds without manager approval."

5

Escalation rules — when to flag for humans?

When should the bot stop and say "I'm escalating this"? "If customer mentions legal action or RBI complaint, immediately say you are escalating."

Key Insight

Same gpt-4o-mini model. Same customer question. 4 completely different responses. The only thing that changed was the system prompt. You're not retraining the model. You're directing it — like a director briefing an actor before a scene.

Important Rule

The system message is sent once, at index 0, in every API call. You don't repeat it in user messages. You don't put it in the prompt text. It's a separate message object with role="system" — and LangChain's ChatPromptTemplate handles this automatically for you.

Session 07 — Chapter 5 of 9

Under the hood.

Chat Completion feels like magic — the model seems to remember, understand context, maintain personality. Let's kill the mystery and see exactly what happens technically from the moment you hit invoke() to the moment text appears on screen.

1

You send a structured message list

Your code calls chain.invoke({"input": "How many EMIs are left?"}). Behind the scenes, LangChain takes your full message history — system + all previous turns + this new message — and packages it as a structured list.

2

SDK serializes it to a flat string

Before sending to the model, the list is flattened into one continuous text block the model can process. Think of it as: System: [rules] | User: [turn1] | AI: [reply1] | User: [new msg]

3

Tokenizer splits everything into tokens

The entire flattened string — every word, number, rupee symbol, period — becomes a sequence of integer token IDs. "BFL2024001" might become 4-5 separate tokens. The model only understands numbers.

4

LLM predicts the reply — one token at a time

The model reads all input tokens, uses its transformer attention mechanism to understand context and relationships, then generates the reply token by token. "You" → "have" → "22" → "EMIs" → "remaining" — each word is predicted, not retrieved.

5

Response wrapped as assistant message and returned

The token sequence is decoded back to readable text, wrapped as {"role": "assistant", "content": "..."}, and returned to your code. You append it to your message list — now it becomes part of the history for the next turn.

Why This Matters for Your App

You are responsible for the conversation history. The model trusts whatever you send it. If your list has incorrect history — model gives incorrect answers confidently. If your list drops a critical turn — model forgets that context instantly. This is why memory management (Buffer, Window, Summary) is a core skill in Session 7.

The Implication — Everything Grows

Since you resend everything every turn, the amount of data you send to the API grows with each message. Turn 10 sends 10× more tokens than Turn 1. This has a direct impact on your API bill — and your app's performance. That's what Context Window (Chapter 7) and Memory Strategies (Chapter 8) are designed to solve.

Session 07 — Chapter 7 of 9

The context window.

Every model has a maximum number of tokens it can read in one request. Everything you send — system prompt, all previous turns, the new message — counts toward this limit. Exceed it, and the model silently forgets the oldest messages.

The Whiteboard Analogy

Imagine the context window as a whiteboard with a fixed size. You write messages on it. When it fills up, you erase the oldest messages to make room for new ones. The model only knows what's currently on the board — nothing that was erased.

The model isn't broken when it "forgets" — it literally cannot see what's been erased from the board.

What Fills the Context Window

Every token in your API call counts toward the limit:

What Happens at Overflow?

Silent Failure

When you exceed the context window, the model silently drops the oldest messages. No error is thrown. The bot suddenly "forgets" the loan number from 20 turns ago. It starts asking questions it already answered. Tone becomes inconsistent. You have no idea why — unless you know about context windows.

Session 07 — Chapter 8 of 9

3 memory types.
Pick the right one.

Since you control the conversation history, you get to decide what to include and what to drop. Three strategies — each with different cost and memory tradeoffs. Understanding when to use which one is a production skill.

Why Memory Strategy = Cost Strategy

Every token you include in the message list = money. A 30-turn Bajaj Finance conversation could be 6,000+ tokens of history alone — resent on every single turn. With 10,000 daily customers, the difference between Buffer and Summary Memory could be thousands of rupees per day in API costs.

Strategy 01

Buffer Memory

Keep every single turn in full detail. Nothing is ever dropped or compressed. The model always has complete context.

Cost: Grows linearly every turn
Context loss: None
Best for: Short sessions (<10 turns), critical context where nothing can be lost
Strategy 02

Window Memory

Keep only the last K turns. Oldest turns are silently dropped. Cost stays fixed regardless of conversation length.

Cost: Fixed — always k messages
Context loss: Early turns dropped
Best for: Long sessions where recent context matters most
Strategy 03

Summary Memory

After every N turns, compress old turns into a concise summary via another LLM call. Keep summary + recent turns.

Cost: Low + summarizer overhead
Context loss: Minimal (key facts preserved)
Best for: Long sessions where details matter

Buffer Memory — Deep Dive

The simplest strategy. InMemoryChatMessageHistory stores every HumanMessage and AIMessage in a Python list. RunnableWithMessageHistory automatically fetches this history, injects it into your prompt's MessagesPlaceholder, and saves the new turn back — all without you writing any glue code.

buffer_memory_explained.py
# InMemoryChatMessageHistory — a simple Python list with helper methods
history = InMemoryChatMessageHistory()
history.add_user_message("Loan BFL2024001 — EMI ₹8,450 deducted twice.")
history.add_ai_message("Logged as High Priority. Refund in 24 hours.")
# history.messages = [HumanMessage(...), AIMessage(...)]
# This ENTIRE list gets injected into your prompt on every turn.

# MessagesPlaceholder — a slot in your ChatPromptTemplate
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a Bajaj Finance support agent..."),
    MessagesPlaceholder(variable_name="history"),  # ← history goes HERE
    ("human", "{input}"),                          # ← new message goes HERE
])

# RunnableWithMessageHistory — automates everything
# Before every .invoke():
#   1. Fetches history from store using session_id
#   2. Injects into MessagesPlaceholder
#   3. Calls the LLM
#   4. Saves new HumanMessage + AIMessage back to store
bajaj_bot = RunnableWithMessageHistory(
    chain,
    get_session_history,    # session_id → InMemoryChatMessageHistory
    input_messages_key="input",
    history_messages_key="history",
)

Summary Memory — How It Works

A separate "summarizer chain" is called every N turns. It takes old messages, compresses them into 2–3 sentences, and stores the summary as a SystemMessage prepended to the recent turns. The main chain sees: [Summary of earlier turns] + [Last 2 full turns] + [New message].

Session 07 — Quiz

Test your
understanding.

8 questions. All based on the Bajaj Finance examples from this session. No tricks — just checking if the concepts landed.

Q1. You send Turn 1 with loan number BFL2024001. For Turn 2, you call llm.invoke("How many EMIs are left?") with no history. What happens?
LLMs are completely stateless. Turn 1 and Turn 2 are separate HTTP requests to the API. The model on Turn 2 has never seen Turn 1. It starts fresh every single time.
Q2. In a Chat Completion message list, which role would you use for: "You are a Bajaj Finance agent. Never mention competitor banks."?
The system role is for behavioral instructions written by the developer. It shapes every assistant response for the entire session without being visible to the user.
Q3. You have a 20-turn Bajaj Finance conversation with Buffer Memory. On Turn 21, how many messages does your API call send?
Buffer Memory sends everything. System + 20 turns × 2 messages (human + AI) + 1 new = 42 messages. That's why Buffer Memory costs grow linearly and why Window/Summary Memory exist.
Q4. What does MessagesPlaceholder(variable_name="history") do inside a ChatPromptTemplate?
MessagesPlaceholder is a slot — a named placeholder. When RunnableWithMessageHistory runs, it takes the history from your store and fills this slot with the past messages before sending to the LLM.
Q5. After a 30-turn conversation, the Bajaj bot suddenly forgets the loan number from Turn 1. No error was thrown. What most likely happened?
When the total token count exceeds the model's context window, the oldest messages are silently dropped — no error, no warning. The model simply cannot see what's no longer in the window.
Q6. Rahul has a 40-turn loan dispute conversation. Which memory strategy should you use to keep costs manageable while ensuring the bot still knows Rahul's original complaint from Turn 1?
Summary Memory is the best fit: it keeps costs low (compresses instead of resending everything) while preserving critical context (Rahul's name, BFL2024001, original complaint details) in the summary.
Q7. You want to run two concurrent Bajaj Finance sessions — one for Rahul (loan BFL2024001) and one for Priya (loan BFL9988). How do you ensure they don't share history?
The session_id in config={"configurable": {"session_id": "..."}} is the key that maps to a specific InMemoryChatMessageHistory. Different session_id = completely isolated history. One bot can handle thousands of concurrent users this way.
Q8. In LangChain v0.3, which component wraps your LCEL chain and automatically fetches, injects, and saves conversation history on every invoke()?
RunnableWithMessageHistory is the automation layer. It wraps any LCEL chain and before every .invoke(): 1) fetches history using session_id, 2) injects into MessagesPlaceholder, 3) calls the LLM, 4) saves the new turn back to the store.
