Chapter 01

Dense · Sparse · Hybrid
Index Types

Three fundamentally different ways to store and search your vectors. Each solves a different problem.

You've been using dense indexes — HNSW, IVF — where every document becomes a 384-dim float vector via all-MiniLM-L6-v2, and retrieval is cosine similarity. This is great for semantic search. Today we add sparse (keyword-level) and hybrid (both).

Dense Index

1.1 Dense Indexes — Capturing Meaning

Dense vectors store every dimension as a meaningful float. A 384-dim vector from MiniLM encodes the full semantic context of a sentence. Two sentences with completely different words but the same meaning will have similar vectors.

✅ Dense Wins When

Customer asks: "Can I defer my EMI payment?" — Dense finds documents about "loan restructuring", "moratorium", "payment holiday" — even though none of those words appear in the query. That's semantic understanding.

Sparse Index

1.2 Sparse Indexes — Keyword Precision

Sparse vectors have mostly zeros — typically 99%+ zero. Only a few dimensions are non-zero, each corresponding to a specific word/token in the vocabulary. Think of TF-IDF or BM25.

🎯 BajajBot Bug Fix

Customer types: "BFL-PL-2024-MUMBAI-007" — a product code. Dense search fails because no embedding model "understands" what this code means. Sparse search (BM25) finds it instantly because it matches exact character sequences.

How BM25 Works — Intuition

BM25 is an advanced TF-IDF. For a query word w in document d:

score(w, d) = IDF(w) × [f(w,d) × (k1 + 1)] / [f(w,d) + k1 × (1 - b + b × |d|/avgdl)]

Where: f(w,d) = word frequency in doc, |d| = doc length, avgdl = average doc length, k1=1.2, b=0.75. The denominator penalizes long documents — so a word appearing 5 times in a 10-word doc ranks higher than 5 times in a 1000-word doc.

📌 Key Characteristics
  • Majority of dimensions are zero (99%+ sparsity)
  • Stored as key-value pairs: {word_position: score}
  • Vocabulary size can be 30,000–100,000 dimensions
  • Exact keyword matching — no semantic approximation
  • Interpretable: you know exactly which word caused the match

Hybrid Index

1.3 Hybrid Indexes — Best of Both Worlds

🕵️ Two Detectives

Think of hybrid search as hiring two detectives. Detective Dense has a "poetic sense" — finds contextually related documents even if words don't match. Detective Sparse has a "photographic memory" — catches exact keywords, codes, and specific terms. Both go search. They return to you with their top suspects. You merge their lists with a weighted score. The best answers rise to the top.

The Alpha Parameter — Key Concept

Hybrid scoring = alpha × dense_score + (1 - alpha) × sparse_score

Alpha controls the balance. This is the most important tunable parameter in hybrid search:

🎛️ Interactive: Tune Alpha

Sparse Dense α=0.5
Balanced: equal weight to semantic meaning + exact keywords. Good starting point.

Hybrid Search in LangChain — EnsembleRetriever

LangChain's EnsembleRetriever is the native hybrid search pattern — combines BM25Retriever (sparse) + ChromaDB (dense) with configurable weights:

# Install: pip install langchain langchain-community langchain-openai chromadb rank_bm25
from langchain_community.retrievers import BM25Retriever
from langchain_community.vectorstores import Chroma
from langchain.retrievers import EnsembleRetriever
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain.chains import RetrievalQA
from langchain.schema import Document

# ─── YOUR BAJAJBOT DOCUMENTS ──────────────────────────────────
bajaj_docs = [
    Document(page_content="Personal loan BFL-PL-2024 processing fee is 2% of loan amount",
             metadata={"source": "faq_pl", "type": "fees"}),
    Document(page_content="EMI deferment or moratorium available during financial hardship",
             metadata={"source": "faq_emi", "type": "policy"}),
    Document(page_content="BFL-PL-2024-MUMBAI interest rate is 10.5% per annum",
             metadata={"source": "faq_rates", "type": "rates"}),
    Document(page_content="KYC documents required: Aadhaar, PAN, 3 months bank statement",
             metadata={"source": "faq_kyc", "type": "documents"}),
    Document(page_content="Can I defer my EMI? Yes, contact helpdesk for loan restructuring options",
             metadata={"source": "faq_defer", "type": "policy"}),
]

# ─── RETRIEVER 1: DENSE (ChromaDB) ───────────────────────────
# Understands MEANING — "defer EMI" ↔ "payment holiday" ↔ "moratorium"
embeddings = OpenAIEmbeddings()   # or HuggingFaceEmbeddings for free/local

chroma_db = Chroma.from_documents(
    documents=bajaj_docs,
    embedding=embeddings,
    persist_directory="./bajajbot_chroma"   # ← saved to disk, no re-encoding on restart
)
dense_retriever = chroma_db.as_retriever(search_kwargs={"k": 5})

# ─── RETRIEVER 2: SPARSE (BM25) ──────────────────────────────
# Understands EXACT WORDS — "BFL-PL-2024", "Aadhaar", product codes
sparse_retriever = BM25Retriever.from_documents(bajaj_docs)
sparse_retriever.k = 5

# ─── COMBINE: EnsembleRetriever (LangChain Hybrid) ───────────
# weights = [sparse_weight, dense_weight]
#
# weights=[0.5, 0.5] → balanced (good default to start)
# weights=[0.3, 0.7] → favor dense  → for conversational queries
# weights=[0.7, 0.3] → favor sparse → for product codes, exact terms

hybrid_retriever = EnsembleRetriever(
    retrievers=[sparse_retriever, dense_retriever],
    weights=[0.3, 0.7]   # 30% BM25 + 70% ChromaDB
)

# ─── TEST: QUERY THAT NEEDS BOTH ─────────────────────────────
query = "What is the interest rate for personal loan BFL-PL-2024?"
# Dense handles: "interest rate", "personal loan" semantics
# Sparse handles: "BFL-PL-2024" exact code match

results = hybrid_retriever.invoke(query)
for i, doc in enumerate(results, 1):
    print(f"{i}. [{doc.metadata['source']}] {doc.page_content}")

# ─── PLUG INTO FULL RAG CHAIN ────────────────────────────────
# The beauty: just swap your retriever — NOTHING else changes!
llm = ChatOpenAI(model="gpt-4o-mini")

qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=hybrid_retriever,    # ← drop-in replacement for any other retriever
    return_source_documents=True
)

response = qa_chain.invoke({"query": query})
print("
🤖 BajajBot:", response["result"])
print("
📄 Sources used:")
for doc in response["source_documents"]:
    print(" -", doc.metadata["source"])

Chapter 02

Vector
Quantization

100 million vectors × 768 floats = 300 GB. That doesn't fit in RAM. Product Quantization compresses vectors by 16–32x with minimal accuracy loss — here's exactly how.

The Scale Problem

2.1 Why Quantization Exists

💣 The Math — Feel the Pain First

BajajBot today: 3,000 FAQ chunks × 384 floats × 4 bytes = ~4.6 MB. No problem.

BajajBot at Bajaj Finance enterprise scale: 100 million document chunks × 768 floats × 4 bytes = ~300 GB.

That won't fit in RAM. You can't load the index. Search breaks. The entire RAG pipeline fails. This is a real production problem.

100,000,000 × 768 × 4 bytes = 307,200 MB ≈ 300 GB 🔥

2.2 The Core Idea — Face Description Analogy

👤 Forget "photo palette" — think face description

Imagine describing someone's face to a sketch artist. You don't say "pixel 1 is RGB(232, 198, 167), pixel 2 is RGB(229, 195, 162)..." — that's 10 million exact values.

Instead you say: "Nose shape: type-23 (from our catalogue of 256 nose shapes). Eye shape: type-07. Jawline: type-118. Skin tone: type-44."

You're picking from a pre-built catalogue. The description is approximate — but good enough to recognize the person. And instead of 10 million numbers, you just stored 4 catalogue IDs.

In PQ terms: each "body part" = a sub-vector chunk. Each catalogue = a codebook with 256 cluster centers. Each catalogue ID = 1 byte.

2.3 Why You MUST Split the Vector — The Key Insight

🧠 The Question Students Always Ask

"Why not just build ONE big codebook for the whole 384-dim vector?"

Answer: Because 256 centroids cannot meaningfully represent a 384-dimensional space. Imagine trying to describe every possible face with just 256 templates — you'd need millions of templates. The approximation error would be catastrophic.

But if you split the face into 96 independent 4-dim sub-spaces, 256 templates per sub-space is actually enough. Each small space is simple enough that 256 centroids cover it well.

ORIGINAL VECTOR (384 dimensions)
V = [0.91, 0.43, 0.05, -0.34, 0.88, 0.12, -0.67, 0.23, ... × 384]
Storage: 384 × 4 bytes = 1,536 bytes
↓ split into 96 chunks of 4 dims each
CHUNK 1
[0.91, 0.43, 0.05, -0.34]
→ centroid 23
CHUNK 2
[0.88, 0.12, -0.67, 0.23]
→ centroid 118
CHUNK 3
[0.33, -0.91, 0.44, 0.07]
→ centroid 7
... × 93 more chunks
↓ store only the centroid IDs
COMPRESSED REPRESENTATION
[23, 118, 7, 201, 44, 88, ...] ← 96 numbers, each 0–255 = 1 byte
Storage: 96 × 1 byte = 96 bytes (was 1,536 bytes → 16x compression)

2.4 The Compression Math — Feel the Win

Original: 384 dims × 4 bytes (float32) = 1,536 bytes per vector After PQ: 96 chunks × 1 byte (uint8) = 96 bytes per vector Ratio: 16x compression ✅ 100M vectors × 1,536 bytes = 300 GB → 100M × 96 bytes = 18 GB 🎉 Fits in a normal server with 32GB RAM!

2.5 The 3 Steps of PQ — Exactly How It Works

StepWhat happensCode equivalent
1. SplitDivide each 384-dim vector into m=96 sub-vectors of 4 dims eachm=96 parameter in FAISS
2. Train codebookRun k-means on each sub-space → 256 cluster centers per sub-spaceindex.train(emb)
3. EncodeReplace each sub-vector with the ID (0–255) of its nearest centroidindex.add(emb)

2.6 FAISS PQ Code — Step by Step

import faiss
import numpy as np

# Simulate embeddings (replace with your actual BajajBot embeddings)
N, d = 10_000, 384   # 10K docs, 384-dim vectors
emb  = np.random.randn(N, d).astype("float32")

# ─── STEP 1: CHOOSE YOUR PQ PARAMETERS ───────────────────────
m     = 96   # number of chunks (d must be divisible by m: 384/96 = 4 ✓)
                # more chunks = better accuracy, slightly more memory
nbits = 8    # bits per chunk → 2^8 = 256 centroids per codebook
                # 8 bits = 1 byte per chunk (standard choice)

# ─── STEP 2: CREATE INDEX ─────────────────────────────────────
index_pq = faiss.IndexPQ(d, m, nbits)
# d=384 dimensions, m=96 sub-quantizers, nbits=8 bits

# ─── STEP 3: TRAIN CODEBOOK ───────────────────────────────────
# k-means runs on each sub-space separately
# Needs at least 256 * m training samples (more is better)
train_size = min(50_000, N)
train_data = emb[np.random.choice(N, train_size, replace=False)]

print("Training codebooks... (builds 96 separate k-means models)")
index_pq.train(train_data)   # ← this is where centroids are learned
print("Training done!")

# ─── STEP 4: ADD VECTORS (stored as compressed codes) ─────────
index_pq.add(emb)   # stores 96 bytes per vector, not 1536 bytes
print(f"Index has {index_pq.ntotal} vectors")

# ─── STEP 5: SEARCH ───────────────────────────────────────────
q     = emb[0:1]    # query: shape (1, 384)
D, I  = index_pq.search(q, k=5)
print(f"Top-5 IDs:       {I[0]}")
print(f"Approx distances: {D[0].round(3)}")
# Distances are APPROXIMATE — that's the trade-off

# ─── MEMORY COMPARISON ────────────────────────────────────────
print(f"
Memory comparison for {N} vectors:")
print(f"  Flat (exact):  {N * d * 4 / 1e6:.1f} MB")
print(f"  PQ (compressed): {N * m * nbits // 8 / 1e6:.1f} MB")
print(f"  Compression: {(d * 4) // (m * nbits // 8)}x")

# OUTPUT:
# Memory comparison for 10000 vectors:
#   Flat (exact):    15.4 MB
#   PQ (compressed): 0.96 MB
#   Compression: 16x

2.7 IVF + PQ — The Production Combo

Use IVF+PQ when your dataset is large. IVF clusters documents into groups — at search time, only the top-N clusters are searched (not all). PQ compresses the vectors inside each cluster.

📌 IVF+PQ Mental Model

IVF = divide a library into sections (Finance, Law, Medical...). When searching, only look in the relevant section — not the entire library.

PQ = within each section, store book summaries instead of full books. Fast to compare, tiny memory.

Together = find the right section (IVF) → compare compressed summaries (PQ). Fast + memory efficient.

# IVF + PQ — the go-to for large-scale production
N, d    = 10_000, 384
m       = 96      # PQ sub-quantizers
nbits   = 8       # 256 centroids per codebook
nlist   = 100     # IVF clusters (rule of thumb: sqrt(N))
nprobe  = 10      # clusters to search at query time ← speed/recall knob
                    # higher nprobe = better recall, slower search

# IVF needs a coarse quantizer (finds nearest cluster centroid)
quantizer    = faiss.IndexFlatL2(d)
index_ivfpq  = faiss.IndexIVFPQ(quantizer, d, nlist, m, nbits)

# Train: IVF centroids + PQ codebooks (needs min 39*nlist samples)
train_size = min(100_000, N)
index_ivfpq.train(emb[np.random.choice(N, train_size, replace=False)])

index_ivfpq.add(emb)
index_ivfpq.nprobe = nprobe    # set BEFORE searching

D, I = index_ivfpq.search(emb[0:1], k=5)
print(f"IVF+PQ Top-5: {I[0]}")

# ─── nprobe TUNING GUIDE ──────────────────────────────────────
# nprobe=1   → fastest, lowest recall (only checks 1 cluster)
# nprobe=10  → good balance (start here)
# nprobe=nlist → same as brute force (defeats the purpose)

🎯 When Should BajajBot Use PQ?

Current state (3K chunks): Don't use PQ. Use IndexFlatIP — exact, fast, tiny memory.
10K–1M chunks: Use IndexHNSWFlat — fast ANN, no compression needed.
>10M chunks or RAM constrained: Use IndexIVFPQ — compression becomes essential.

Chapter 03

Evaluating
Retrieval Quality

You've built the retrieval pipeline. But is it actually working? These metrics tell you whether BajajBot is fetching the RIGHT chunks — not just any chunks.

🎬 The Problem

BajajBot is returning results. But your PM asks: "Are you sure it's fetching the right documents?" You can't just say "it feels right". You need numbers. That's what retrieval metrics are for.

Imagine BajajBot searched "vegetarian loan products Mumbai" and returned: [1] Random insurance doc, [2] Personal Loan Mumbai ✓, [3] Home Loan ✓, [4] Credit Card, [5] Auto Loan. Even though 2 relevant docs appear, the first result was wrong — and users only look at the top result.

Order-Unaware

3.1 Precision@K and Recall@K

These metrics only care whether relevant results appear in top-K, not where they are.

Precision@K

Of the K results returned, what fraction were actually relevant?

P@K = (relevant in top-K) / K

If top-5 has 3 relevant docs → P@5 = 3/5 = 0.60

Recall@K

Of ALL relevant docs in the corpus, how many did we find in top-K?

R@K = (relevant in top-K) / (total relevant)

If 8 relevant total, 3 in top-5 → R@5 = 3/8 = 0.375

Order-Aware

3.2 MRR, MAP, NDCG — Order Matters

In real systems, rank-1 is gold. Rank-5 is already disappointing. These metrics penalize relevant results that appear lower in the list.

MetricWhat It MeasuresBest For
MRR@KWhere does the FIRST relevant doc appear? Score = 1/rankWhen only the top answer matters
MAP@KAverage precision at each relevant doc rank — rewards putting many relevants earlyMulti-answer queries
NDCG@KDiscounted gain — logarithmically penalizes lower ranks, normalized to 0-1Graded relevance (some answers more relevant than others)
📐 Worked Example — All Metrics

Top-5 results: [A, B, C, D, E] | Relevant set: {B, E, G} (3 total relevant docs)

  • Precision@5 = 2/5 = 0.40 (B and E are in top-5)
  • Recall@5 = 2/3 = 0.67 (found 2 of 3 relevant docs)
  • MRR@5 = 1/2 = 0.50 (first relevant is B at rank 2)
  • AP@5 = (0.5 + 0.4) / 3 = 0.30 (precision at rank 2 + rank 5, divide by 3 total relevant)
  • NDCG@5 = DCG / IDCG < 1.0 (sub-optimal because B and E aren't at top)

3.3 Full Implementation — All Metrics

import math

# ─── METRIC FUNCTIONS ─────────────────────────────────────────

def precision_at_k(ranked_ids, relevant_ids, k):
    """Of top-K results, what fraction are relevant?"""
    topk = ranked_ids[:k]
    hits = sum(1 for x in topk if x in relevant_ids)
    return hits / max(1, k)

def recall_at_k(ranked_ids, relevant_ids, k):
    """Of all relevant docs, how many appear in top-K?"""
    if not relevant_ids: return 0.0
    topk = ranked_ids[:k]
    hits = sum(1 for x in topk if x in relevant_ids)
    return hits / len(relevant_ids)

def mrr_at_k(ranked_ids, relevant_ids, k):
    """Mean Reciprocal Rank: 1/rank_of_first_relevant"""
    for i, x in enumerate(ranked_ids[:k], start=1):
        if x in relevant_ids:
            return 1.0 / i
    return 0.0

def average_precision_at_k(ranked_ids, relevant_ids, k):
    """Average precision — rewards putting ALL relevants early"""
    if not relevant_ids: return 0.0
    hits, ap = 0, 0.0
    for i, x in enumerate(ranked_ids[:k], start=1):
        if x in relevant_ids:
            hits += 1
            ap   += hits / i    # precision at this rank
    return ap / len(relevant_ids)   # divide by TOTAL relevant (not just found)

def dcg_at_k(ranked_ids, gains_dict, k):
    """Discounted Cumulative Gain — logarithmically penalizes lower ranks"""
    dcg = 0.0
    for i, x in enumerate(ranked_ids[:k], start=1):
        g = gains_dict.get(x, 0.0)        # 0 if not relevant
        dcg += g / math.log2(i + 1)       # discount by log2(rank+1)
    return dcg

def ndcg_at_k(ranked_ids, gains_dict, k):
    """Normalized DCG — compare to ideal ranking. Score: 0–1"""
    dcg  = dcg_at_k(ranked_ids, gains_dict, k)
    ideal = sorted(gains_dict.items(), key=lambda kv: kv[1], reverse=True)
    idcg = dcg_at_k([d[0] for d in ideal], gains_dict, k)
    return 0.0 if idcg == 0 else dcg / idcg

# ─── BAJAJBOT EVALUATION EXAMPLE ──────────────────────────────
# Query: "What is the processing fee for personal loan?"
ranked    = ["faq_credit_card", "faq_personal_loan_fee",
             "faq_home_loan", "faq_pl_docs", "faq_emi_calc"]
relevant  = {"faq_personal_loan_fee", "faq_pl_docs", "faq_emi_calc"}
gains     = {"faq_personal_loan_fee": 3,  # most relevant
             "faq_pl_docs": 2,            # somewhat relevant
             "faq_emi_calc": 1}           # marginally relevant

k = 5
print(f"Precision@{k}:  {precision_at_k(ranked, relevant, k):.3f}")
print(f"Recall@{k}:     {recall_at_k(ranked, relevant, k):.3f}")
print(f"MRR@{k}:        {mrr_at_k(ranked, relevant, k):.3f}")
print(f"AP@{k}:         {average_precision_at_k(ranked, relevant, k):.3f}")
print(f"NDCG@{k}:       {ndcg_at_k(ranked, gains, k):.3f}")

# Expected output (approx):
# Precision@5:  0.600
# Recall@5:     1.000  ← all 3 relevant found in top-5
# MRR@5:        0.500  ← first relevant at rank 2
# AP@5:         0.511
# NDCG@5:       0.734  ← not 1.0 because best doc isn't rank-1

Chapter 04

QPS &
Latency

Retrieval quality is only half the story. In production, BajajBot needs to handle thousands of concurrent users without slowing down. That's QPS and latency.

4.1 QPS — Queries Per Second

QPS measures how many queries your system can process per second across a measurement window. It depends on hardware, index type, top-k size, efSearch, and concurrency.

⚠️ QPS Alone is Misleading

"Our system handles 200,000 QPS!" sounds great. But if 1% of queries take 2 seconds, you have a real problem — those are millions of failed requests per day at scale.

4.2 Latency Percentiles — p50 / p95 / p99

Never trust the average. In production, the worst 1% of queries define your system's stability.

MetricValueMeaning
QPS150,000Great throughput
p50 (median)5 msTypical user waits 5ms — excellent
p9550 msSlowest 5% wait 50ms — acceptable
p991,200 ms 🔴1% of users wait >1s — causes timeouts!

Even with great QPS and p50, a high p99 kills production. Frontend timeouts, cascading delays in RAG pipelines, SLA breaches.

    hnsw_idmap.search(q_mat[i:1+i], top_k)

# ─── SINGLE-THREAD BASELINE ───────────────────────────────────
def measure_single_thread(q_vectors):
    lat_ms = []
    start  = time.perf_counter()
    for qv in q_vectors:
        t0 = time.perf_counter()
        hnsw_idmap.search(qv.reshape(1, -1), top_k)
        lat_ms.append((time.perf_counter() - t0) * 1000)
    total = time.perf_counter() - start
    lat_ms = np.array(lat_ms)
    return {
        "mode":          "single-thread",
        "total_queries": len(q_vectors),
        "total_time_s":  total,
        "qps":           len(q_vectors) / total,
        "p50_ms":        float(np.percentile(lat_ms, 50)),
        "p95_ms":        float(np.percentile(lat_ms, 95)),
        "p99_ms":        float(np.percentile(lat_ms, 99)),
    }

# ─── CONCURRENT + BATCHED ─────────────────────────────────────
def measure_concurrent(q_vectors, max_workers=8, batch_size=8):
    batches = [q_vectors[i:i+batch_size]
               for i in range(0, len(q_vectors), batch_size)]
    lat_ms = []
    start  = time.perf_counter()

    def task(batch):
        t0 = time.perf_counter()
        hnsw_idmap.search(batch, top_k)    # matrix search — all at once
        per_q_ms = ((time.perf_counter() - t0) * 1000) / len(batch)
        return per_q_ms, len(batch)

    with ThreadPoolExecutor(max_workers=max_workers) as ex:
        futures = [ex.submit(task, b) for b in batches]
        for fut in as_completed(futures):
            per_q_ms, n = fut.result()
            lat_ms.extend([per_q_ms] * n)

    total  = time.perf_counter() - start
    lat_ms = np.array(lat_ms)
    return {
        "mode":          f"workers={max_workers} batch={batch_size}",
        "qps":           len(q_vectors) / total,
        "p50_ms":        float(np.percentile(lat_ms, 50)),
        "p95_ms":        float(np.percentile(lat_ms, 95)),
        "p99_ms":        float(np.percentile(lat_ms, 99)),
    }

# ─── RUN & COMPARE ────────────────────────────────────────────
s = measure_single_thread(q_mat)
print(f"\n[SINGLE THREAD] QPS={s['qps']:.0f} p50={s['p50_ms']:.2f}ms p99={s['p99_ms']:.2f}ms")

for workers in [4, 8]:
    for bsz in [1, 8, 32]:
        c = measure_concurrent(q_mat, workers, bsz)
        print(f"[{c['mode']}] QPS={c['qps']:.0f} p50={c['p50_ms']:.2f}ms p99={c['p99_ms']:.2f}ms")

Chapter 05 · New Concept

Bi-Encoder vs
Cross-Encoder

You've been using bi-encoders since Session 2 without knowing it. Reranking (Chapter 6) uses cross-encoders. Understanding the difference is why reranking works.

⚠️ This was never introduced in the course

Every time you called model.encode("What is my EMI?") — that's a bi-encoder. Every time Pinecone BGE Reranker runs — that's a cross-encoder. Same word "encoder", completely different architectures. Let's fix that gap right now.

5.1 The Fundamental Difference

🔵 Bi-Encoder

Encodes query and document separately and independently. They never "see" each other. Score = cosine similarity of two vectors.

query → [Encoder] → vector q
doc → [Encoder] → vector d
score = cosine(q, d)

They are scored independently — never compared side by side.

🟢 Cross-Encoder

Reads query + document together as one input. Model sees full interaction. Outputs a single relevance score.

[query + " [SEP] " + doc]
→ [Encoder]
→ relevance score (0–1)

Full interaction — model reads both and judges them together.

5.2 The Hiring Analogy — Clearest Way to Explain

🏢 Job Interview Analogy

Bi-Encoder = Resume Screener
HR reads 10,000 resumes independently. Each gets a score from 1–10 based on keywords and experience. Fast — one person can do 100 resumes per hour. But they never directly compare two resumes against each other. They never ask "which of these two candidates is better for THIS specific role?"

Cross-Encoder = Panel Interview
The final 10 candidates come in. The panel asks questions specific to the role. The candidate and the job requirements are in the room together — the panel can probe gaps, test fit, compare directly. Slow — takes 1 hour per candidate. But far more accurate than a resume screen.

RAG Pipeline = Both stages together. Bi-encoder screens 10,000 chunks down to 100. Cross-encoder interviews those 100 and picks the top 5 for the LLM.

5.3 Why Bi-Encoder is Fast but Limited

The bi-encoder's magic is pre-computation. At indexing time, every document is already encoded into a vector and stored. At query time, only the query needs encoding. Then it's just a dot product — incredibly fast even for millions of documents.

⚡ The Speed Secret

Indexing time (done once): encode all 3,000 BajajBot chunks → store vectors in ChromaDB.

Query time (every request): encode 1 query → dot product with 3,000 stored vectors → done in milliseconds.

But the limitation: The model never sees query and document together. It's like rating two restaurants without ever eating at either one — just looking at their menus separately.

5.4 Why Cross-Encoder is Slow but Precise

The cross-encoder cannot be pre-computed. For every query, it must re-process every (query, document) pair from scratch. You can't encode the document alone — it always needs the query present.

🎯 The Precision Secret

The cross-encoder reads: "What is my EMI? [SEP] EMI for loan BFL2024001 is ₹8,450 per month, deducted on the 5th."

It can see: "EMI" appears in both. "BFL2024001" is a specific loan ID. The document directly answers the question. Score = 0.97 (very relevant).

A bi-encoder only knows "EMI query" and "EMI document" are semantically close — it can't see the specific loan ID match. Score = 0.81 (less precise).

Vector search returns top-100 candidates. But the truly best answer might be at rank 40. Reranking fixes that with a smarter second pass.

Retrieval = swiping through 10,000 profiles and picking 100 that "kinda match" your preferences (fast, broad).

Users only look at the first 3-5 results. If rank-1 is wrong, your RAG system fails — even if the correct answer was at rank 8.

5.2 BM25 Reranking — Full Code

import re, math, numpy as np, faiss
from sentence_transformers import SentenceTransformer

# ─── SETUP ────────────────────────────────────────────────────
model    = SentenceTransformer("all-MiniLM-L6-v2")
# texts, doc_ids, emb_norm, hnsw_idmap from previous sessions
id_to_pos = {int(ids_int64[k]): k for k in range(len(ids_int64))}

# ─── BM25 SETUP ───────────────────────────────────────────────
def tokenize(s: str):
    return re.findall(r"[a-z0-9]+", s.lower())

corpus_tokens = [tokenize(t) for t in texts]
doc_lens      = np.array([len(t) for t in corpus_tokens], dtype=np.float32)
avgdl         = float(doc_lens.mean()) if len(doc_lens) else 1.0
df            = {}
for toks in corpus_tokens:
    for w in set(toks): df[w] = df.get(w, 0) + 1

N_docs = len(corpus_tokens)
k1, b  = 1.2, 0.75

def idf(w):
    n = df.get(w, 0)
    return math.log((N_docs - n + 0.5) / (n + 0.5) + 1.0)

def bm25_score(query_tokens, doc_idx):
    toks = corpus_tokens[doc_idx]
    if not toks: return 0.0
    tf   = {}
    for w in toks: tf[w] = tf.get(w, 0) + 1
    dl   = len(toks)
    norm = k1 * (1 - b + b * dl / avgdl)
    score = 0.0
    for w in query_tokens:
        f = tf.get(w)
        if not f: continue
        score += idf(w) * (f * (k1 + 1)) / (f + norm)
    return score

# ─── ANN RETRIEVAL (Stage 1) ──────────────────────────────────
def ann_retrieve(query_text: str, top_k: int = 100):
    q  = model.encode([query_text], convert_to_numpy=True).astype("float32")
    faiss.normalize_L2(q)
    D, I = hnsw_idmap.search(q, top_k)
    positions = [id_to_pos[int(rid)] for rid in I[0] if int(rid) in id_to_pos]
    scores    = D[0][:len(positions)].tolist()
    return positions, scores

# ─── BM25 RERANK (Stage 2) ────────────────────────────────────
def bm25_rerank(query_text: str, candidates, top_k: int = 10):
    qtoks  = tokenize(query_text)
    scored = [(bm25_score(qtoks, i), i) for i in candidates]
    scored.sort(reverse=True)
    return scored[:top_k]

# ─── FULL PIPELINE: BEFORE vs AFTER ──────────────────────────
query = "processing fee for personal loan BFL-PL-2024"

candidates, vec_scores = ann_retrieve(query, top_k=100)

print("=== BEFORE RERANKING (ANN top-5) ===")
for rank, (pos, sc) in enumerate(zip(candidates[:5], vec_scores[:5]), 1):
    print(f"{rank}. [{doc_ids[pos]}] sim={sc:.4f} | {texts[pos][:80]}...")

reranked = bm25_rerank(query, candidates, top_k=10)

print("\n=== AFTER BM25 RERANKING (top-5) ===")
for rank, (bm25, pos) in enumerate(reranked[:5], 1):
    print(f"{rank}. [{doc_ids[pos]}] bm25={bm25:.4f} | {texts[pos][:80]}...")

5.3 Pinecone BGE Reranker — Cross-Encoder (Best Quality)

BM25 reranking is fast and interpretable. But for maximum quality, use a cross-encoder like BGE Reranker — it reads query + document together and gives a deep relevance score.

from pinecone.grpc import PineconeGRPC as Pinecone

pc    = Pinecone(api_key=api_key)
index = pc.Index("bajajbot-hybrid")

id_to_text = {str(doc_ids[i]): texts[i] for i in range(len(doc_ids))}

def stage1_retrieve(query_text: str, top_k: int = 25):
    """Stage 1: Fast ANN retrieval from Pinecone"""
    qv  = model.encode([query_text], normalize_embeddings=True)[0].tolist()
    res = index.query(vector=qv, top_k=top_k,
                      include_values=False, include_metadata=False)
    return [(m["id"], float(m["score"])) for m in res.get("matches", [])]

def stage2_rerank(query_text: str, candidates, top_n: int = 5):
    """Stage 2: Cross-encoder reranking via Pinecone BGE Reranker"""
    docs = [
        {"id": _id, "text": id_to_text.get(str(_id), "")}
        for _id, _ in candidates
        if id_to_text.get(str(_id))
    ]
    if not docs: return []
    rr = pc.inference.rerank(
        model="bge-reranker-v2-m3",     # cross-encoder model
        query=query_text,
        documents=docs,
        top_n=top_n,
        return_documents=True
    )
    return [(item.document["id"], float(item.score), item.document["text"])
            for item in rr.data]

# ─── RUN FULL 2-STAGE PIPELINE ────────────────────────────────
query = "What is the interest rate for personal loan BFL-PL-2024?"

candidates = stage1_retrieve(query, top_k=25)
print("=== Stage 1: ANN Retrieval (top-5) ===")
for rank, (_id, score) in enumerate(candidates[:5], 1):
    print(f"{rank}. {_id} | sim={score:.4f}")

reranked = stage2_rerank(query, candidates, top_n=5)
print("\n=== Stage 2: BGE Reranker (top-5) ===")
for rank, (_id, score, text) in enumerate(reranked, 1):
    print(f"{rank}. {_id} | rerank={score:.4f} | {text[:80]}...")

6.1 Query Caching

🏦 BajajBot Reality

Every day, 10,000 customers ask BajajBot: "What is the minimum balance for savings account?" Without caching, that's 10,000 embedding calls + 10,000 vector searches + 10,000 LLM generations. With caching: compute once, serve 9,999 times instantly.

Simple In-Memory Cache with Semantic Match

import hashlib
import numpy as np
from functools import lru_cache

# ─── EXACT MATCH CACHE (simplest) ─────────────────────────────
query_cache = {}   # {query_hash: (top_k_results, timestamp)}

def cache_key(query: str) -> str:
    return hashlib.md5(query.lower().strip().encode()).hexdigest()

def cached_retrieve(query_text: str, top_k: int = 5):
    key = cache_key(query_text)
    if key in query_cache:
        print("✅ Cache HIT")
        return query_cache[key]
    # Cache MISS — do the real work
    q  = model.encode([query_text], convert_to_numpy=True).astype("float32")
    faiss.normalize_L2(q)
    D, I = hnsw_idmap.search(q, top_k)
    results = [(doc_ids[id_to_pos[int(rid)]], float(sc))
               for rid, sc in zip(I[0], D[0]) if int(rid) in id_to_pos]
    query_cache[key] = results
    print("❌ Cache MISS — computed and stored")
    return results

# ─── SEMANTIC CACHE (catches near-duplicates) ─────────────────
# Store (embedding, result) pairs; for new queries, check cosine sim
# If similarity > threshold → return cached result (even if wording differs)

cache_embeddings = []   # list of (embedding, result) tuples
SIMILARITY_THRESHOLD = 0.92   # tune based on your domain

def semantic_cached_retrieve(query_text: str, top_k: int = 5):
    q = model.encode([query_text], convert_to_numpy=True,
                     normalize_embeddings=True).astype("float32")[0]

    # Check against cached embeddings
    for cached_emb, cached_result in cache_embeddings:
        sim = float(np.dot(q, cached_emb))   # cosine sim (both normalized)
        if sim >= SIMILARITY_THRESHOLD:
            print(f"✅ Semantic cache HIT (sim={sim:.3f})")
            return cached_result

    # MISS — compute and cache
    results = cached_retrieve(query_text, top_k)
    cache_embeddings.append((q, results))
    return results

# Test: near-duplicate queries should HIT cache
semantic_cached_retrieve("minimum balance savings account")      # MISS
semantic_cached_retrieve("what is minimum balance in savings account") # HIT!

6.3 FAISS vs Pinecone — Production Comparison

🔧 FAISS (Library)

  • No built-in caching — implement yourself (Redis, dict)
  • Batching: manual — you manage index.search(batch)
  • Scaling: DIY — sharding, load balancing, merge logic
  • Monitoring: none — build your own Prometheus metrics
  • Repeat queries: always re-scans the index
  • Memory growth: manual eviction + index rebuild

☁️ Pinecone (Managed)

  • Built-in query caching at infrastructure level
  • Automatic batch optimization per request
  • Auto-scaling: serverless, no manual sharding
  • Dashboard: QPS, p50/p95/p99, upsert rate built-in
  • Repeat queries: served from hot cache
  • Data: live upserts, instant availability
🎯 Decision Rule for BajajBot

Development / Learning: Use FAISS. Full control, no cost, perfect for understanding internals (Sessions 3–4).

Production at scale: Use Pinecone. Caching, monitoring, scaling — all handled. You focus on the RAG logic, not infra.

Hybrid approach: FAISS for local dev + testing. Pinecone for staging + prod. Same query code, just swap the index object.
